# ADR 0003: Use UUIDs for Chat Channels

## Status
Approved

## Context
Currently, a Chat Channel between two users is defined implicitly by combining their email addresses (e.g., `mary@imagey.cloud:alice@imagey.cloud`). This approach has several drawbacks:
1. It exposes contact information (email addresses) in URLs and APIs.
2. It assumes a 1-to-1 relationship, making it difficult to support multiple channels per user-pair (or group chats) in the future.
3. If users change their email addresses, the channel identity breaks.

Furthermore, contacts were stored in a folder structure (`[user]/contacts/[contact-email]/key.json`), and the API `GET /contacts` returned a list of `User` objects, requiring a separate request for the encryption key.

## Decision
1. We will transition to using a UUID to uniquely identify a `Channel`. 
2. Channels will be generated by the backend when a contact invitation is accepted.
3. The storage mechanism for a contact will change from a folder to a single JSON file (`[user]/contacts/[contact-email].json`).
4. This contact JSON will contain the contact user details, the `channelId`, and the encryption `key`.
5. The `GET /contacts` API will return a list of these `Contact` JSON objects instead of `User` objects, reducing the number of requests the frontend needs to make.

## Consequences
- The URL structure on the frontend will change from `/chats/:contactEmail` to `/chats/:channelId`.
- The `ContactRepository` in the backend will be refactored to read/write `Contact` JSON files instead of managing folders.
- The `ContactKeys` or `ContactService` will generate a `Channel` UUID upon accepting an invitation.
- The `GET .../contacts/{contact}/key` endpoint might become redundant but can be kept if necessary for specific key reissuance flows.
- Frontend's `Contact` type and `contactRepository.getContacts()` will be updated to handle the new JSON structure.
